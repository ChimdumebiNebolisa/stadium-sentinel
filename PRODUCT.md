# Product

## Register

product

## Users
Venue operations staff working live soccer stadium events. They need to move from one messy report to a clear incident response workflow without losing priority, ownership, or supporting context.

## Product Purpose
Stadium Sentinel is an incident operations command center. It turns a raw event report into prioritized incidents, response actions, team dispatch, evidence, timeline updates, staff wording, and a post-event report.

Success means operators can answer these questions quickly:
- Which incident needs attention first?
- What team should respond now?
- What evidence, timeline, and report artifacts support the response?

## Binding Direction
- Stadium Sentinel is a soccer stadium incident operations product.
- The main desktop UI is a 2-section command center: left dispatch queue, right active incident workspace, bottom utility drawer.
- The main workflow is messy report to incident split to priority to response workflow to team dispatch to evidence, timeline, and report.
- The stadium map is no longer the main product surface.
- Full seat-map packages, seat-map libraries, seat selection, ticketing UI, ticket purchase flows, and seat chart editors are out of scope and will not be implemented.
- Soccer stadium terminology remains the binding venue context for copy, seeded data, and UI framing.

## Layout Default
- Top command bar: Stadium Sentinel, Riverside Stadium, Live operations, open incident count, top priority.
- Left dispatch queue: incident selection only.
- Right active incident workspace: incident header, action row, checklist, team assignment, timeline summary, evidence feed.
- Bottom utility drawer: Evidence, Staff Update, Timeline, Report.

## Non-Goals
- No map-first composition.
- No seat-map product direction.
- No ticketing or purchase workflows.
- No CRM-style admin dashboard framing.
- No numeric scoring, confidence meters, or severity scores in the product UI.

## Brand Personality
Focused, credible, operational. The tone should read as calm under pressure, direct, and compact without clutter.

## Design Principles
- Make the right workspace dominate the page.
- Give each region one job: queue selects, workspace executes, drawer supports.
- Avoid repeating the same incident summary across multiple regions.
- Keep the interface dense but readable, with restrained accents and minimal noise.
- Show operational context and next action without turning the product into a map demo.

## Accessibility & Inclusion
Target strong desktop readability, visible focus states, stable layouts, clear contrast, and predictable interactions for staff scanning quickly during live event operations.
